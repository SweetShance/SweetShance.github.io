3、django 的信号机制

Django 的信号机制允许在 Django 的不同部分之间传递信息。当某些事件发生时，信号会触发，并调用相应的处理函数。

核心概念：

发送者（Sender）：触发信号的对象（如Model、Request等）

信号（Signal）：特定事件的通知（如保存、删除等）

接收者（Receiver）：监听信号并执行相应操作的函数

调度器（Dispatcher）：管理信号和接收者的注册关系

pre_save 信号在保存模型实例之前触发。
post_save 信号在保存模型实例时触发。
pre_delete 信号在删除模型实例之前触发。
post_delete 信号在删除模型实例时触发。
pre_init 信号在初始化模型实例之前触发。
post_init 信号在初始化模型实例时触发。
m2m_changed 信号在 ManyToManyField 字段值改变时触发。
pre_migrate 信号在迁移操作之前触发。
post_migrate 信号在迁移操作之后触发。

迭代器：
1.是一个带状态的对象，在你调用 next() 方法时，会返回下一个元素。任何实现了 iter() 和 next() 方法的对象都可以被迭代。
2. 他不会把元素一次性加载到内存中而是，在需要的时候进行加载。
生成器：
1.不通过return 返回而是通过 yield 返回的， 所有生成器一定是迭代器，但并非所有迭代器都是生成器。
生成器的列表推导示： (x*x for x in range(10))

yield 的作用是暂停函数，并返回一个值。当调用next 的时候，会从 yield 的位置继续执行，直到再次遇到 yield 或 return。

请解释Python中列表（List）的内存分配策略，并编写代码演示列表的动态扩容机制。

要求：

解释列表的过度分配（over-allocation）策略

演示列表扩容时的内存增长模式

对比 append()、extend()、insert()操作的内存行为

提供优化建议

核心设计：
内存连续：元素存储在连续的 内存块中
随机访问：列表的元素可以通过索引快速访问
列表的动态扩容机制：当空间不足时自动分配更大的内存块
过度分配策略：
每次满园的时候不止增加一个二十按照 几何级数 进行扩容 0 4 8 16 25 35

  1. append() 操作
    最佳情况：容量足够，直接追加，O(1)时间复杂度
    最坏情况：需要扩容，复制所有元素到新数组，O(n)时间复杂度
    分摊分析：平均每次append仍是O(1)
  2. insert(0, item) 操作
    需要将所有元素向后移动，O(n)时间复杂度
    可能触发扩容，性能极差
    推荐使用collections.deque
  3. extend() 操作
    一次性扩容到所需大小，避免多次扩容
    比多个append()更高效

提供字典内存优化的实际建议

示例代码框架：

  1. 字典分为 索引数组和条目数组，索引数据存的是hash值与条目数组的对应关系，条目数组按插入顺序存储 键值对
    存储逻辑：首先将key进行hash，如果该hash值已存在，则使用开放寻址法，寻找空闲位置。
    取值逻辑： 通过hash值找到索引位置，然后通过索引位置找到条目数组中的键值对，如果值 不一样，采用开放寻址接着寻找
  1. 字典的扩容机制：
    按当前大小的2倍或者4倍进行扩容，扩容细节是：
    分配新的索引数组和条目数据，重新计算key的hash值，将旧的护具复制到新的数据组中，将旧的内存回收。

列表的话因为是索引数组，所以是连续的，所以存储 紧凑，内存效率高，而字典的话会有大量的空桶，内存效率低，扩容频率上来说，列表扩容频率高但是增量小，字典不频繁但是增量大。

提供字典内存优化的实际建议： 在存储的时候使用更紧凑的键，增加内存使用效率
